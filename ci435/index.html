Setting up brighton domains

The lab session started with logging into the computers and accessing the MyStudies page. Here I could go to CI435 to view all the information about the web development module.

In the module area there are the following sections:

  • Module Information which shows the module specification, the teaching plan and link to teaching staff contact details

Starting at the induction and going through setting up brighton.domains. I went to the dashboard tab at the top and logged into my account.

Here I can access the file manager to see the files hosted on the server.

We were instructed to create a new folder named ‘ci435’ under the public_html folder which is the root folder of the hosted website.

With how the domain hosting is setup, I would create a new subdomain for each project so rather than a subdirectory called ci435, I would make a subdomain ci435.jb2139.brighton.domains. This separation would make for future package management easier should I use wordpress or another content management system.

Week 2: Marking up content

Published on October 14th, 2020

This week our objective was to 1:

  1. Add a publication date to each post
  2. Add a list to a post
  3. Create in-page navigation links
  4. Add a copyright-free image

Adding publication dates

The publication date uses the HTML5 element <time> . I checked the Mozilla documentation for the time element and the datetime attribute is a machine-readable format that improves search engine results.2

I added this new publication date to each article in their <header> elements and updated both the display text as well as the datetime attribute. I will look into adding a colour to these elements in the future to ensure they stand out.

Unordered and ordered lists

Next I used an unordered list <ul> at the top of the first week post for the module sections and an ordered list <ol> at the end of the first week post for I learned in our first week. Each list uses an encompassing tag with list items <li> within to represent each point or number.

I also used the <code> tag to cover the code elements and <em> tags for document names and other important information to emphasize their importance. These elements look a bit plain however I will add more styling to them later on.

Navigation links using anchor tags

Next I added a relative link using the same <a> tag by instead of linking to another local site page or an external site page, I added a link to an ID of an element on the same page.

I did this by making the anchor’s href link to an element’s ID. I added an anchor element at the top of the page and gave it the ID of #top. By clicking the link, the browser automatically scrolls to the element with the matching ID.

Since each ID can/should only be used on a single element, the page should never scroll to the wrong element.

Adding images and copyright

After, I had to add a copyright-free image to the weekly posts using the <figure>, img and figcaption tags.

The <figure> tag is an image wrapper that illustrates the web page content. Within this element is contained the <img> tag which is the source file of the image. Also within the figure element is the <figcaption> tag which illustrates the source image attached.

To ensure the picture was copyright-free, I took a picture of the Huxley building from my iPhone 7 and used that.

I added at least a single image to all posts thus far to later be used for banner images. Intend for the website to look its best on all possible screen sizes including mobile and high resolution desktops.

Note that I have temporarily added a style block in the header to make images sane for a while since their intended sizes cannot be dynamically be set yet due to no CSS file, no viewport or media queries.

Creating email anchor tags

For the email link, I added another anchor tag to my footer with a href that includes the mailto flag to open the user’s default mail application and send me an email.

To conclude this week’s tutorial I validated my code on w3.org3 again to ensure I hadn’t made any silly mistakes and that I am conforming to the HTML5 standard. Sure enough it came back with no errors!

Further learning

I was determined to save myself some time by adding automation for deployment and looked into if GitHub actions could, rather than use FTP to send the updated content, use Rsync to sync the files from the GitHub account over to the remote hosting server.

The server is hosted on a Linux Apache machine, I could use my credentials to SSH into the machine. Sure enough Rsync is available on the box and could upload files that way. I then looked into the GitHub Action to deploy the code and I found this one:3

This looked promising. I unfortunately couldn’t create a deploy user to ensure if anything happened to my GitHub account, they wouldn’t gain full system access however I used secrets for all my credentials and upload destinations to ensure security.

Finally I had my code uploading without a hitch and I could say goodbye to FileZilla eating into my time having to constantly upload the new files.

To compare time to deploy between the FTP and RSYNC, FTP took 1 minute on average and didn't delete old removed files. RSYNC completes in 20 seconds even with large payloads and removes the unused files to match the structure in the repository. Definitely a big upgrade!

I also did some research into the pre tag3 as in future posts, I would very much like to add full blocks of code rather than inline. The pre tag enclosing the code tag like so

<pre><code>
    System.out.println();
</code></pre>

This way with some future formatting, I could still use the code semantics with a wrapper to ensure it was on its own line with the pre-formatted code maintained!

Make use of nested lists

We have previously used lists in week 2 however we can nest a list within a list to create a sub list. <ul> and <ol> only allow exactly zero or more <li> elements however we can use an <li> element and add a list to it.

We can also create a third type of list called a description list. Like the other lists, we use an encapsulating tag <dl> (description list) and within we have description term <dt>. We also have the term description <dd> which is used to further expand on the term with a long description. This would be useful for something like a glossary of terms with the term as a <dt> and an <dd> containing its description.

Adding a table

Back to the point, we can create a table using the table tag and within it, add table rows tr, table headers th for the headers and table data td for the table cells.

Since tables don't have any styling, we need to add border="1" in order to add a border around each cell and the table as a whole. This I know has been deprecated in favour of using CSS to style it. After all , HTML should be the structure containing the content and the CSS only for the styling.

Adding normalize.css from GitHub

This week we are focusing on Cascading Style Sheets or CSS for short. It describes how HTML elements are to be displayed on different mediums such as screens, paper or even brail.

The first stylesheet I am adding is the normalize.css stylesheet from Necolas on GitHub. This stylesheet has a very useful function. It ensures that all the basic elements are displayed in a consistent manor, or 'normalize', as the name would suggest.

The external stylesheets are added in the <header> element at the top of the HTML document. This would look something like this:

<link href="css/normalize.css" rel="stylesheet" />

Adding our own stylesheet

Now the page is normalized, I can add my own new stylesheet for my own layout and colour schemes. For this I created another new file called stylesheet.css in the css folder. Here I used the template provided by the tutorial which contained some empty blocks where styling can be added.

Style the body element

The body element is the first viewable element of the page and represents the whole viewable content. By setting a background colour using background-color: gray; we can set the background colour to a nice grey.

Style other elements

The headings h1 - h6

These header elements are used in a similar way to to a report. The first heading, heading 1, represents the top level with all following in a natural hierarchy. For this purpose we can resize all of these elements in descending order.

The paragraph <p> tag

The paragraph tag is initially influenced or inherited by the css rules set in the body. I decided to add the Montserrat font from Google as the primary font for all content on my web page. I initially set it for the p tags however founding it more fitting to simply add it in the body style since it will affect list items and other non p elements too.

The wrapper <div id="wrapper">

The wrapper element we are using to encompass all elements on the page, we can assign style to that container. As a start, having all the content in a max size box would make reading far easier and having that content centered would be more visually pleasing composition wise. To do this, I set the max-width value to 1000px and set the margin to auto. This way the wrapper will shrink down but only expand up to 1000px. And when there is a gap, add a margin to both sides equally.

I will later be removing this wrapper as this can easily be done using the body element and use css grid to organize content within. I will want some banner images to span the full screen width in the future too. By creating a max width and margin now, I will have trouble spanning an image across.

The anchor <a> tag pseudo-classes

The anchor elements have different states depending on what actions have been taken on the page. For example, when the anchor tag has a link, it goes from being the regular font colour to having a link colour.

On top of that, if you hover over the link, the style changes again to use yet another colour and add an underline to the text.

We can affect these states by referencing the <a> tag's pseudo-elements. These are represented as a with a colon and the pseudo-element name. For example, the one for hover would look like this:

a:hover {
    color: red;    
}

In this example, we have every anchor element which is hovered, change its colour to red.

There is an order to these pseudo-elements and must be updated in a certain order. This being: a, a:link, a:visited, a:hover

We can also influence only certain tags within other elements to make, for instance, only links within the articles red instead of blue which is applied everywhere else. To do this, I added the following:

article a:link {
    color: red;
}

By adding the parent element and then the element we want to affect, we can specify exactly where we want these css rules to apply.

The main menu <nav>

Navigation menus are usually implemented using list items as they contain natural hierarchy with nesting. The issue is that the elements aren't placed side-by-side left to right and they have bullet points next to each element.

We can remove these by setting the list-style-type to none and make the nav's anchor elements have a different link style.

The article headers <header class="post">

I could apply styles directly to the header however this would affect both the article header and the page header at the same time. Instead I applied the styling to the post class which I can specify on all the article headers.

A more efficient method for doing this would be to simply state 'only apply to headers which are children of the article element' like this:

article header {
    background-color: grey;
}

This would prevent the requirement of each header needing the class set and adding another article would be trivial.

As always, the last part of testing is to check the HTML and CSS is HTML5 compliant. Neither of these documents resulted in any errors and is ready to be uploaded!

Viewport Meta Tag

The viewport meta tag in the head of the document is used to help control the width and scaling of the viewport so that the content is sized correctly on all devices.

External Fonts

I have already started using google fonts from last week and have now added 2 more fonts; one for the main h1 headings and the other for code blocks to use a monospace font.

Floating Images

Block elements are usually placed one below the other. In some cases we may want content to wrap or be beside other content such as text around an image. For this we need to use the float style on an element.

Line Height, Text Indenting and Alignment

These are very useful styles which we have control over. Line height affects the distance between lines of the same paragraph and can improve readability for those who are hard of seeing or suffer like I do from learning disabilities such as dyslexia.

Text indentation alters the indent at the beginning of a paragraph and can shift the text along without the need to add random spaces to the start of the text (which would be very bad practice).

Text Alignment allows the control of how the text is formatted to the page. By default the text is left aligned however we can change this to be right aligned, centered or even justified.

This week our objectives were to 1:

  2. Setup a responsive layout

Adjusting HTML and CSS

I updated the layout of articles to use the banner in its own container. This is preparation for media queries to adjust the design.

Setup Responsive Layouts

I have updated the images to fill their containers using width: 100%;. With this and a max-width: 800px; for the article containers, I can now make images a useful size on mobiles and desktops.

Implementing a Mobile-First Approach

I have started this design primarily for the standard mobile size most used in the market. With more devices accessing the internet from small size devices like mobiles and other handheld touch devices, I have my design, before the use of media queries, to be for mobiles first.

Using Media Queries

By using @media queries, I can set different CSS rules to be applied based on the screen size. I have setup a couple of initial checks for devices smaller than the original design to cater for tiny screens like iPod touch generations and other old iPhones which have a 4.7" screen.

Next I have added more rules for devices larger than a standard mobile aimed at tablet devices which are still used with touch input.

Finally I have a general rule for screens that are the standard screen size for small laptops 1024px up to any size as the text will scale.

Hiding Elements

For the smallest screen sizes, I have chosen to hide all the image content as the main content does not require the images within <figure>

Testing the Design

For testing, I have fired up my old iPod Touch, iPhone 7 and my recent phone iPhone 11 to give a good range of mobile screen sizes. I also checked on my mobile app development Huawei P20 to ensure android devices don't have any weird issues. I also checked across different mobile browsers. All showed the same result thanks to the normalize.css code and my stylesheet.css layout.

Week 7: Flexible media

Published on November 18th, 2020

This week our objectives were to 1:

  1. Make the media (images and video) on the page flexible
  2. Optimise images for better loading times
  3. Refactor code accordingly
  4. Run validation

Making Media Flexible

For the most part, I have set my images that are within the <figure> tags to be set to width: 100%; on the main layout for mobile. This way all images will be as clear as possible without much scrolling required.

I have also set the banner images to be background elements when moving to a desktop size screen. This has been achieved using CSS Grid with the scroll element set to sticky to lock it in place.

Having all these images on the page is now affecting the loading speed of the page. I have been using an extension made by Google to rank the page on design for accessibility, speed and performance. This extension is called Lighthouse and was primarily developed to aide in the effort of creating mobile first websites along with Progressive Web Apps (PWA).2

A PWA website is able to be installed on a device and run locally without requiring internet. This is very powerful for developers to make a website that can double as an application that users can bookmark or even install to their mobile devices and use like a native application.

I have attached the score given to my site so far and will show the improved performance after having refactored the code and optimized the images in the next sections.

Optimizing Images

The main cause for such a poor performance score is due to the images and more to the point, their file sizes. There are many ways to reduce an image's file size. The route I took is to run them through a compressor service such as compressor.io. This website will accept images up to 10 megabytes in size and compress the images to shrink the file size. I have applied this to all of the images used on the site bar one which is the logo used in the hero section. I want this logo to be in the highest quality possible to be eye catching.

From simply reducing these file sizes using a compressor, the results speak for themselves:

Further Optimization

The fun for optimization doesn't stop there. We can also reduce the dimensions of the image as the full resolution of the images are not always going to be seen. For this, we can put our images through an image editor and reduce the resolution to be the maximum width that it will ever be displayed in.

For my site, I'm using a column design that alternates between left and right. This benefits me in that I never need a full screen sized image as these panels lock out at 960px each. I do have to be careful as the mobile-first design can also stretch to 1200px before the styles switch.

The final result of reducing all the images to this max displayable size has further affected the loading score!

On a final note, there is a last optimization to the css style for each of the banner images. Here I set the mobile first design to use a half-sized image when on mobile to save bandwidth and for desktops to use the full fat image as these will most likely be on wired internet rather than mobile data. Just a small nudge to help mobile users.

Refactoring Code

Making my banner image appear as a background required a new div element with the banner images set as a background-image so it would fall behind the content and not be clickable. I have also taken this chance to restructure and comment all my css stylesheet code into discrete sections that can easily be followed and rules found. I use a :root block for core colours that can be referenced throughout the sheet as primary colours, page colours, hover transitions times and so on. This way if I wish to change the primary colour, I only need to update the root value and the change will be referenced everywhere.

Validation

With the refactoring of code, I wanted to make sure there were no errors or validation errors introduced. Thankfully since all the code was simply moved and small changes updated, there were no errors in either the HTML or CSS code.
