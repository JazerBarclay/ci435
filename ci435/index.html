Week 1: Brighton Domains Setup

Published on October 7th, 2020

For my first week of CI435, I was tasked with reading our course specification on MyStudies and setting up my brighton.domains account where I will be making my first three HTML pages.1

The lab session started with logging into the computers and accessing the MyStudies page. Here I could go to CI435 to view all the information about the web development module.

In the module area there are the following sections:

  • Module Information which shows the module specification, the teaching plan and link to teaching staff contact details
  • Reading List - A list of books we can access virtually to learn more about HTML and CSS
  • Assessments and Marks - A grading criteria and module guideline
  • Study Materials - Where all the content from the lectures is accessible and other learning resources

In the study section, there’s a folder called 'SEMESTER 1 study material' where the slides for the previous lectures are available.

Under week 1 there is a link to the lecture and another for the lab tutorial site jh1033.bright.domains with instructions for building the website.

Starting at the induction and going through setting up brighton.domains. I went to the dashboard tab at the top and logged into my account.

Here I can access the file manager to see the files hosted on the server.

We were instructed to create a new folder named ‘ci435’ under the public_html folder which is the root folder of the hosted website.

Since this is an apache server hosted on a linux system, the folders when we later access them on the live URL, are case sensitive so keeping to lower case makes navigation simple as long as we keep to that design pattern. Note also that the web page names must also follow this convention.

In this new folder we can create a new file named ‘index.html’. This is the default file the server will use when the folder is accessed. Here is where our learning journal site will begin.

There was a template provided in the lab tutorial which we were instructed to copy into the contents of the index.html document and save.

With the contents saved, if I navigated to the url jb2139.brighton.domains/ci435, I could see the copied template live from any device.

Next I created the second and third pages for the tutorial and contact pages respectively using the same template from the first (index) page.

These were based from the same template however must link back to all there other pages on the site.

Though these were based on the same template, they must link back to all the other pages on the site.

I used the <a> anchor tag to create links between the different pages and ensured they all worked on every page.

I found that the live version of the site sometimes didn’t update and as a result I found that Chrome has ‘hard refresh’ functionality where holding Shift and clicking the refresh button would force the page to be called rather than using a cached version which was no longer up to date.

As a final step to our first week lab session, we needed to test that our code conformed to the W3C standards for HTML5.

My website index.html, tutorial.html and contact.html all gave no errors and no warnings.

I checked the next week’s lab page and we will be covering HTML markup such as time, lists, anchors, images and email links.2

I notices when writing up this section that there references are in an aside element only in the wrapper rather than per section. I did some searching for if the HTML5 standard recommends only a single aside element however using one per article is accepted as good practice. The aside element is used to identify content that is related to the primary content of the webpage, but does not constitute the primary content of the page.3

Since testing the web pages, I have added my code to Github where all changes are stored and kept backed up. By hosting my code here, I can keep track of all the weekly changes and revert my code should I break something and want to go back to the last working version.4

Downloaded the code from GitHub to my personal laptop where I could make local changes without constantly updating the web page. I’ve found this useful when working off-line when commuting and also prefer the interface of Visual Studio Code which is the editor I use for programming.5

Furthermore I have started to use FileZilla to upload my code since the GitHub action cannot upload automatically due to how the SFTP server is configured.6

In summary, this week I learned how to:

  1. Set up a folder for my website on my Brighton Domains workspace
  2. Link two pages together with a navigation menu using a relative link
  3. Create links to external websites using an absolute link
  4. When to use the h1 and h2 tags
  5. Update the web page title
  6. Use the W3C validator to validate HTML documents

Week 2: Marking up content

Published on October 14th, 2020

This week our objective was to 1:

  1. Add a publication date to each post
  2. Add a list to a post
  3. Create in-page navigation links
  4. Add a copyright-free image
  5. Create an email link

The publication date uses the HTML5 element <time> . I checked the Mozilla documentation for the time element and the datetime attribute is a machine-readable format that improves search engine results.2

I added this new publication date to each article in their <header> elements and updated both the display text as well as the datetime attribute. I will look into adding a colour to these elements in the future to ensure they stand out.

Next I used an unordered list <ul> at the top of the first week post for the module sections and an ordered list <ol> at the end of the first week post for I learned in our first week. Each list uses an encompassing tag with list items <li> within to represent each point or number.

I also used the <code> tag to cover the code elements and <em> tags for document names and other important information to emphasise their importance. These elements look a bit plain however I will add more styling to them later on.

Next I added a relative link using the same <a> tag by instead of linking to another local site page or an external site page, I added a link to an ID of an element on the same page.

I did this by making the anchor’s href link to an element’s ID. I added an anchor element at the top of the page and gave it the ID of #top. By clicking the link, the browser automatically scrolls to the element with the matching ID.

Since each ID can/should only be used on a single element, the page should never scroll to the wrong element.

After, I had to add a copyright-free image to the weekly posts using the <figure>, img and figcaption tags.

The <figure> tag is an image wrapper that illustrates the web page content. Within this element is contained the <img> tag which is the source file of the image. Also within the figure element is the <figcaption> tag which illustrates the source image attached.

To ensure the picture was copyright-free, I took a picture of the Huxley building from my iPhone 7 and used that.

I added at least a single image to all posts thus far to later be used for banner images. Intend for the website to look its best on all possible screen sizes including mobile and high resolution desktops.

Note that I have temporarily added a style block in the header to make images sane for a while since their intended sizes cannot be dynamically be set yet due to no CSS file, no viewport or media queries.

For the email link, I added another anchor tag to my footer with a href that includes the mailto flag to open the user’s default mail application and send me an email.

To conclude this week’s tutorial I validated my code on w3.org3 again to ensure I hadn’t made any silly mistakes and that I am conforming to the HTML5 standard. Sure enough it came back with no errors!

I was determined to save myself some time by adding automation for deployment and looked into if GitHub actions could, rather than use FTP to send the updated content, use RSYNC to sync the files from the GitHub account over to the remote hosting server.

The server is hosted on a Linux Apache machine, I could use my credentials to SSH into the machine. Sure enough rsync is available on the box and could upload files that way. I then looked into the GitHub Action to deploy the code and I found this one:3

This looked promising. I unfortunately couldn’t create a deploy user to ensure if anything happened to my GitHub account, they wouldn’t gain full system access however I used secrets for all my credentials and upload destinations to ensure security.

Finally I had my code uploading without a hitch and I could say goodbye to FileZilla eating into my time having to constantly upload the new files.

To compare time to deploy between the FTP and RSYNC, FTP took 1 minute on average and didn't delete old removed files. RSYNC completes in 20 seconds even with large payloads and removes the unused files to match the structure in the repository. Definately a big upgrade!

I also did some research into the pre tag3 as in future posts, I would very much like to add full blocks of code rather than inline. The pre tag enclosing the code tag like so

<pre><code>
    System.out.println();
</code></pre>

This way with some future formatting, I could still use the code semantics with a wrapper to ensure it was on its own line with the pre-formatted code maintained!
